# UK Flood Insurance Catastrophe Model

A probabilistic CAT model for UK residential flood risk built from publicly available data. The goal is to produce a credible Loss Exceedance Curve (LEC) and Annual Average Loss (AAL) for a residential portfolio, validated against observed insured losses from UK flood events since 2000.

---

## Why this project exists

This project is designed as an **insurance-focused ML and data engineering system**, not just a notebook model.

- Build a transparent UK flood-cat view from open data that can support pricing, portfolio steering, and risk communication.
- Reproduce insurer-relevant outputs (AAL, LEC, tail risk) and calibrate them against real UK loss history.
- Create a modular platform where new data sources and modelling assumptions can be tested quickly through an AutoResearch loop.

---

## Intended use

This repo is intended for:

- **Insurance analytics and cat teams** who want a reproducible open-data baseline.
- **ML/data engineers** building asynchronous geospatial risk pipelines.
- **Researchers** exploring how UK public datasets can be combined into practical flood-loss estimation.

It is best used as a **decision-support and experimentation platform**, not a regulatory filing model.

---

## What this project does

I've built this as a full four-layer catastrophe model following the structure used in commercial cat modelling:

```
Hazard → Exposure → Vulnerability → Loss Aggregation
```

- **Hazard** — GEV distributions fitted to NRFA annual maximum flows for ~880 gauging stations, giving return period flows at T=2 through T=1000 years
- **Exposure** — ~5M Land Registry transactions geocoded to lat/lon via postcodes.io, with EA flood zone assignment and property value aggregation by postcode
- **Vulnerability** — DEFRA FD2320 residential depth-damage curve with adjustments for contamination and flood duration
- **Loss** — 10,000 stochastic Monte Carlo events producing a full LEC, AAL, and TVaR (99.5%)

The data pipelines are built to be resilient and high-throughput (parallel API extraction, caching, fallback logic), and the project includes an **AutoResearch experiment loop** that can run and evaluate model/data variants against `val_score`.

There's also a Thames RDS scenario benchmarked against the Lloyd's £6.2bn industry loss figure, climate-adjusted AALs under UKCP18 RCP8.5, and a TIV accumulation table by flood zone and county.

Recent additions include:
- **Revised ABI event dataset** with updated 2024 insured loss figures and a new 2025 event entry
- **EA flood pipelines updated to current endpoints** (WFS for flood map layers + flood-monitoring API for warning areas)
- **Hull 2007 case-study pipeline** (EA open data via CKAN API), used as a UK-specific vulnerability calibration anchor

## Data

All data is publicly available and free. The pipeline scripts download and cache everything automatically.

| Source | Used for |
|---|---|
| NRFA (CEH) | Peak flow records for hazard layer |
| HM Land Registry Price Paid | Property values and postcode exposure (2020–2024) |
| EA Flood Map for Planning (WFS) + EA Flood Monitoring API | Flood zone layers and flood warning area polygons |
| VOA Council Tax Stock | Property count by local authority |
| IMD 2019 (MHCLG) | Deprivation index as vulnerability modifier |
| EA Real-Time Monitoring API | Gauging station metadata |
| ABI / DEFRA flood events | Historical observed events (including revised 2024 and added 2025 entries) for model validation |
| EA Hull 2007 open dataset (CKAN) | Property-level/postcode-level empirical flood impact data for vulnerability calibration |
| NFIP Claims (FEMA) | US claims data used as damage function calibration reference |

**Note on EA flood zones:** this project now targets the post-2025 EA data services (WFS + flood-monitoring API). The remaining challenge is not endpoint availability, but producing robust postcode-level risk assignment with strong spatial QA and fallbacks when any upstream service is degraded.

## Running it

```bash
pip install -r requirements.txt

# Download all data (cached, re-runnable)
python run_pipelines.py

# Build geocoded exposure portfolio (~20 min, parallel geocoding)
python src/exposure/portfolio.py

# Run model — prints val_score on last line
python train.py
```

### AutoResearch workflow (optional)

```bash
# Initialize experiment tracking file
python autoresearch/results_init.py

# Run iterative experimentation loop (as described in autoresearch/program.md)
python train.py > run.log 2>&1
grep "^val_score:" run.log
```

## Known limitations and what's needed to fix them

**The biggest gaps right now:**

1. **Postcode-level flood zone assignment quality** — polygon ingestion is now implemented, but assignment quality/coverage at postcode level is still the highest-impact improvement area for tail accuracy.

2. **TIV is inflated** — I'm using Land Registry transactions as a proxy for property count, which overcounts. The portfolio currently holds ~£2,380bn TIV across 5M transactions; realistic residential TIV for England is around £400–600bn. The fix is to use VOA council tax stock counts as the property count denominator rather than transaction count.

3. **No DEM** — flood depth is estimated from a simplified Manning's-law approximation rather than actual elevation data. OS Terrain 50 is free under PSGA and would substantially improve depth estimation.

4. **Flat LEC at high return periods** — the percentage-of-portfolio-flooded formula caps at around T=85 years, so T=100/200/500 losses are currently identical. This is a calibration issue in `train.py` that needs fixing once flood zone assignment is working.

5. **Spatial correlation** — a single scalar factor is used across the whole portfolio. Proper event footprints (flood extent polygons per event) are needed for realistic tail loss behaviour.

6. **Hull 2007 integration depth** — the Hull dataset is now ingested, but calibration can be improved by tightening postcode matching, validating depth distributions, and explicitly linking Hull-derived signals to vulnerability parameter tuning.

**On methodology:**

The hazard layer uses GEV MLE fitted station-by-station. The industry standard for UK hydrology is FEH regional pooling (pooled growth curves across hydrologically similar stations), which gives more stable parameter estimates for stations with short records. I've stubbed this in as `AMAX_REGIONAL_POOLING` in `train.py` but haven't implemented it yet.

The DEFRA FD2320 damage function is the standard for UK residential flood damage assessment, so that part is fit for purpose. The depth offset and contamination parameters are calibrated to minimise val_score against the 13 known events — but there aren't enough events to do a proper train/test split, so the validation is effectively in-sample.

Hull 2007 is used as a targeted UK case study to anchor vulnerability assumptions with empirical post-event data (depth and damage patterns), reducing dependence on non-UK proxies for the most important calibration region.

A proper commercial implementation would link each property to a river reach, run a hydraulic model to get flood extents per return period, and apply the damage function property by property. What I've built is a portfolio-level approximation that captures the right order of magnitude but can't produce spatially granular results.
